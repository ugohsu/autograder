# 名簿（氏名・学籍番号）バッチインポート用 仕様書（AI向け）

あなたはこれから、教員から渡された「氏名・学籍番号 手書き画像一覧」（この採点支援アプリからダウンロードした
PNG画像。人数が多い場合は複数枚に分割されています）を見て、各受験生の氏名と学籍番号を特定し、`roster` という
JSON配列の形式で出力する作業を行います。教員が学生名簿（正解データ）もあわせて渡してくれた場合、これは
単純な手書き文字の書き起こしではなく、**手書きに最も近い名簿上の人物を特定するマッチングタスク**になります
（詳細は後述の「学生名簿が提供された場合の読み取り方針」を参照）。

画像は1行が1人の受験生に対応しており、各行は次の3列で構成されています。

- 左（`No.`列）: 通し番号（`#1`, `#2`, ... のように表示）
- 中央（`Name (handwritten)`列）: 手書きの氏名
- 右（`Student ID (handwritten)`列）: 手書きの学籍番号（1桁ずつマスに書かれています）

画像が存在しない行（読み取りエラーなどで手書き画像自体が無い受験生）は `(no image)` と表示されています。

このJSONは人間が手で書くものではなく、あなたのようなチャット型AIが画像を見て読み取った内容から作成し、
教員がアプリの画面に貼り付けてインポートするためのものです。

## やること

1. 教員から、氏名・学籍番号の手書き画像一覧（複数枚のPNGがある場合はそのすべて）を受け取る
2. （あれば）教員が用意した学生名簿（クラス名簿・出席番号順リストなど、フォーマットは問わない）も受け取る
3. 画像に写っている受験生1人につき1要素、`roster` 配列の要素を作る
   （画像左端に印字されている通し番号を、そのまま `seq` として使うこと。複数枚のPNGにまたがっていても
   通し番号は画像間で重複しない前提なので、気にせずそのまま使ってよい）
4. `(no image)` の行や、手書き文字が不鮮明で自信が持てない箇所があれば、`_comment` に理由を書き、
   `_confidence` を下げる（全く読めない場合は `name` や `student_id` を省略してよい。省略した項目は
   教員が画面上で確認・手入力する）
5. 最終的に `{"roster": [...]}` という1つのJSONオブジェクトを出力する
   （前後に説明文をつけず、JSONのみ、または明確にコードブロックで囲って出力すること）

## 学生名簿が提供された場合の読み取り方針（最重要）

学生名簿が提供された場合、その名簿は**手書きよりも信頼できる正解データ**です。このときの作業は
「手書き文字をそのまま書き起こす」タスクではなく、**「手書きの氏名・学籍番号に最も近い名簿上の要素を
探し出し、その名簿に書かれている正しい表記を採用する」というマッチングタスク**だと考えてください。

- 手書きが多少崩れていたり、学籍番号の1桁が判読しづらくても、名簿の全候補と照合すれば一意に絞り込める
  ことが多い（例: 学籍番号の6桁目が「6」か「8」か迷っても、名簿にその2択のどちらかしか実在しなければ
  実在する方を採用してよい）
- `name`・`student_id` は、最終的に**名簿に記載されている表記のまま**出力する（手書きの崩れやクセを
  反映したり、名簿の表記を手書きの見た目に寄せたりしない）。手書きは「名簿のどの行を指しているか」を
  特定するための手がかりとして使う
- 名簿の中に、確信を持って対応付けられる要素が見つからない場合（候補が複数残る、手書きが不鮮明すぎる、
  名簿に該当しそうな人物がいない、など）は、**推測で無理に1つに決めない**こと。その場合は次のいずれかにする
  - 一番近そうな候補を採用しつつ `_comment` に迷った理由（候補が複数あった、名簿のどれとも決め手に
    欠けた、など）を書き、`_confidence` を低くする
  - まったく決め手がない場合は `name`・`student_id` を省略し、`_comment` にその旨を書く
- 名簿が提供されなかった場合は、これまで通り画像だけを頼りに、手書きの見た目に忠実な書き起こしを行ってよい
  （この場合の `_confidence` は「文字としての読み取りやすさ」を表す）
- 名簿の中に、どの手書き行にも対応しなさそうな要素が余る（欠席者などで手書き画像自体が存在しない）のは
  正常な状態なので気にしなくてよい。逆に、名簿のどれにも対応しない手書き行がある場合
  （転入生・名簿に無い受験生など）は、無理に名簿の値を当てはめず、手書きの読み取り結果をそのまま出力し、
  `_comment` に「名簿に該当者が見つからなかった」旨を書く

## 出力フォーマット

| フィールド | 必須 | 説明 |
|---|---|---|
| `seq` | **必須** | 画像左端に印字されている通し番号（`#1` なら `1`）の整数 |
| `name` | 任意 | 氏名。名簿がある場合は名簿上の正しい表記、無い場合は手書きの見た目に忠実な書き起こし。特定できなければ省略してよい |
| `student_id` | 任意 | 学籍番号。名簿がある場合は名簿上の正しい表記、無い場合は手書きの読み取り結果。**数字だけとは限らないため、必ず文字列として出力すること**（例: `"0123456789"`。先頭のゼロを保持し、数値型にはしないこと。将来アルファベットを含む学籍番号を扱う可能性もある） |
| `_comment` | 任意 | 読み取り・名簿とのマッチングで自信が持てない箇所、判断に迷った点などを自由記述。データベースには保存されず、教員が確認する際の参考情報としてのみ使われる |
| `_confidence` | 任意（既定`1.0`） | 0.0〜1.0の数値。名簿がある場合は「名簿の対応付けにどれだけ確信が持てたか」、名簿が無い場合は「文字としてどれだけはっきり読めたか」を表す。確信が持てない場合はそれに応じて下げる（目安: 0.8未満だと確認画面で警告表示される） |

## 注意事項

- `seq` は同じ画像セット内で重複してはいけない（1人につき1要素）
- `seq` に対応する受験生がアプリ側に存在しない場合、その要素はインポート時にエラーになる。画像に写っている
  通し番号をそのまま使う限り、通常はこの心配は不要
- このタスクでは既存の氏名・学籍番号は渡されない。上書きするか新規登録するかはアプリ側が自動判定するので、
  あなたは気にしなくてよい。ただし、同じ通し番号を再度インポートすると、その受験生の氏名・学籍番号は
  **新しい値で置き換えられる**点に注意（`name` や `student_id` を省略した場合、その項目は空欄で
  上書きされる。まだ確認できていない項目まで空にしたくない場合は、そのまま前回の読み取り値を再度出力すること）
- `student_id` は必ず文字列（ダブルクォートで囲む）で出力すること。数値として出力すると先頭のゼロが
  失われる
- 出力は妥当なJSON（末尾カンマなし、ダブルクォート使用、コメントなし）にすること
- `id` や `batch_id` のようなデータベース内部のIDは、あなたが知る・指定する必要は一切ない

```json
{
  "roster": [
    {
      "seq": 1,
      "name": "試験 太郎",
      "student_id": "2024001234",
      "_confidence": 1.0
    },
    {
      "seq": 2,
      "name": "山田 花子",
      "student_id": "2024005678",
      "_comment": "学籍番号の6桁目が '6' か '8' か判読しづらかったが、名簿には '2024005678' しか実在しないため、こちらで確定した。",
      "_confidence": 0.9
    },
    {
      "seq": 3,
      "name": "佐藤 次郎",
      "_comment": "名簿の候補が '佐藤 次郎' と '佐藤 二郎' の2名で絞り込みきれず、手書きに近い方を暫定的に採用。要確認。学籍番号欄は判読不能のため省略。",
      "_confidence": 0.4
    },
    {
      "seq": 6,
      "_comment": "この受験生は画像が (no image) となっており、手書き画像自体が存在しない（読み取りエラー）。",
      "_confidence": 0.0
    }
  ]
}
```
